# Workout App — Product Plan

From "personal app with one hardcoded program" → "multi-user product where Claude generates personalized programs from a chat".

---

## 1. Vision

A workout PWA where any user can:

1. **Chat with an AI coach** that asks about goals, experience, schedule, injuries, equipment.
2. **Receive a structured N-week program** generated by Claude, built from a known exercise library.
3. **Run that program** with the existing logger (sessions, sets, photos, units, notes, skip, etc.).
4. **Switch between programs** they've created — multi-program support.
5. **Iterate on a program** through the chat ("make week 3 harder", "swap the chest day for a push day").

The current app is the engine. We need: program library, chat-driven program creation, backend, multi-user auth, polish.

---

## 2. Current State (as of 2026-05-12)

**Frontend**: React 18 + Vite + TS + Tailwind, deployed on Firebase Hosting (`boostmind-b052c.web.app/workout-app/`).
**Backend**: None. Pure Firebase + Firestore.
**Auth**: 4-digit passcode → `user_NNNN` UID. No email/identity. Open Firestore rules.
**Programs**: One hardcoded `PROGRAM` in [`src/data/program.ts`](src/data/program.ts) — `program-v1`, 12 weeks, 5 days/week, 6 exercises/day, 3 phases (tempo + rest scaling).
**Library**: ~100 exercises in [`src/data/exerciseLibrary.ts`](src/data/exerciseLibrary.ts), 10 muscle categories, mapped to free-exercise-db images.
**Persistence (Firestore, per user)**:
- `sessions` — `{ date, completedAt, day, weekNumber, phase, programName, completed, partial, sets, skippedExerciseIds }`
- `exerciseStats` — last/max/avg per exercise
- `customExercises` — user-added or overridden
- `hiddenExercises` — per-day hidden IDs
- `exerciseNotes`, `exerciseDifficulty`, `photos`
- `settings/main` — `programStartOverride`

**Features already shipped** (relevant to product transition):
- Hebrew + English bilingual UI
- Unilateral (left/right) split with imbalance protocol
- Skip exercise, tag exercise (alt/pair), unit picker (kg/lb/free text), session date editor, program-start override
- Rest timer with minimize, photo capture per exercise, exercise difficulty rating
- Per-week navigation, partial completion, restart exercise/day, finish early

**Gaps for product readiness**:
1. Single hardcoded program — no creation flow, no multi-program.
2. No backend → can't call Claude.
3. No real auth → can't onboard outside users.
4. Open Firestore rules → not safe to publish.
5. UI polish for first-time users (onboarding, empty states, error states).
6. No billing / abuse limits if Claude is in the loop.

## 3. Target Architecture

```
┌─────────────────────────┐         ┌──────────────────────────┐
│   React PWA (client)    │ ──auth─▶│  Firebase Auth           │
│   - Program creator UI  │         └──────────────────────────┘
│   - Workout logger      │                       │
│   - Program switcher    │ ──data──▶┌──────────────────────────┐
│                         │          │  Firestore               │
│                         │          │  users/{uid}/...         │
│                         │          │  programs/{programId}    │
│                         │          └──────────────────────────┘
│                         │
│                         │ ──chat──▶┌──────────────────────────┐
│                         │          │  Cloud Functions (Node)  │
│                         │          │  /api/programs/chat      │ ──▶ Claude API
│                         │          │  /api/programs/generate  │ ──▶ Claude API
└─────────────────────────┘          │  Library catalog server  │
                                     │  Rate limiting / auth    │
                                     └──────────────────────────┘
```

**Why a server is required**: Claude API key can't live in the browser. Server also: rate-limits per user, validates structured outputs against library, caches the catalog prompt for cost.

**Recommended stack**: **Firebase Cloud Functions** (Node 20). Reasons:
- Same Firebase project — no new vendor.
- Auto-scales, pay-per-call.
- Built-in Firebase Auth verification (`context.auth.uid`).
- Can write Firestore directly from inside the function.
- Easy to deploy with existing `firebase deploy`.

Alternative: a small Node/Express on a VPS (matches the BoostMind monorepo style). Cheaper at high volume, more ops.

## 4. Data Model Changes

### New: `Program` collection

Store under `users/{uid}/programs/{programId}` (per-user, isolated):

```ts
type Program = {
  id: string;                    // stable random ID
  userId: string;
  name: string;                  // user-facing, e.g. "Push/Pull/Legs — Chest Focus"
  nameHe?: string;
  description?: string;
  goals: string[];               // ["maintain chest", "fix left scapula", ...]
  experienceLevel: 'beginner' | 'intermediate' | 'advanced';
  daysPerWeek: number;
  weeksTotal: number;
  phases: Phase[];               // same shape as today
  days: Day[];                   // same shape as today
  startDate: string;             // ISO; today's `program.startDate`
  createdAt: number;
  updatedAt: number;
  isActive: boolean;             // exactly one per user
  isArchived: boolean;
  sourceChatRef?: string;        // chat doc id used to generate
  version: number;               // for iterative edits
};
```

### New: `chats` collection

Persists the program-creator conversation, so users can resume / iterate / re-generate.

```ts
type ProgramChat = {
  id: string;
  userId: string;
  programId?: string;            // set once a program is generated
  messages: { role: 'user' | 'assistant'; content: string; ts: number }[];
  status: 'in-progress' | 'generated' | 'abandoned';
  createdAt: number;
  updatedAt: number;
};
```

### Modified: `Session` schema

Replace `programName: string` with `programId: string`. Backfill existing sessions.

### Modified: `useProgram` / `getAutoWeek`

These currently import `PROGRAM` from `program.ts`. Switch to:

- `useActiveProgram(uid)` — hook that loads the user's active program from Firestore.
- `getAutoWeek` keeps its signature (takes startDate string).
- All consumers (Home, Workout, SessionDetail) read program data from this hook instead of importing `PROGRAM`.

### Hebrew-only naming

Existing types have `name` + `nameHe`. For v1 (Hebrew-only product):
- New programs/exercises generated by Claude populate Hebrew text in `name`. `nameHe` is left empty/deprecated for new content.
- UI display logic: prefer Hebrew (`nameHe` if present, else `name`).
- Existing English/bilingual data (program-v1) keeps both fields — no breaking change.
- The exercise library is rebuilt to be Hebrew-primary: each library entry must have a Hebrew name; the English `name` becomes the secondary/internal identifier.

### Library remains client-side (for now)

`EXERCISE_LIBRARY` stays in [`src/data/exerciseLibrary.ts`](src/data/exerciseLibrary.ts) — it's static catalog data. The server only needs a copy of it to feed to Claude and validate IDs. Both server and client read from the same source (consider extracting to a shared package or syncing via a script).

## 5. Chat-Driven Program Creation

### Flow

```
1. User taps "Create program"
   → Frontend creates a new `ProgramChat` doc
   → Navigates to <ProgramCreatorChat /> screen

2. ChatBot opens with a system-generated first message:
   "Hi! Let's design a program. First — what's your goal?"

3. User replies; frontend POSTs to /api/programs/chat with full message history.

4. Server:
   - Verifies Firebase Auth token
   - Checks rate limit (e.g. 50 messages/day, 5 program-creations/month)
   - Loads conversation, appends user message
   - Calls Claude with:
     - System prompt (workout-program-designer persona, constraints)
     - Cached prompt: full EXERCISE_LIBRARY + MUSCLE_CATEGORIES
     - Tool: submit_program(program: ProgramJSON)
     - Conversation history
   - Returns Claude's reply (text + any tool calls)
   - Persists message to Firestore

5. Claude asks clarifying questions one at a time:
   - "How many days per week can you train?"
   - "Any injuries?"
   - "Equipment access — full gym, home dumbbells, bodyweight?"
   - "Hebrew or English exercise names?"

6. Once Claude has enough info, it calls `submit_program` with a complete JSON
   that conforms to the Program schema. The schema forces every
   exercise.libraryId to be one of the library IDs.

7. Server validates:
   - Every libraryId exists in EXERCISE_LIBRARY
   - Days numbered 1..daysPerWeek
   - Phases cover [1..weeksTotal] without gaps or overlaps
   - sets >= 1, reps >= 1 (or durationSeconds for time-based)
   - All required fields present

8. On valid: server creates Program in Firestore, returns programId.
   Frontend navigates to <ProgramPreview /> screen.

9. <ProgramPreview /> shows the full program, day by day, with library
   images. User can:
   - Accept → set isActive=true, navigate to Home
   - Iterate → return to chat, "change Tuesday to a pull day"
   - Discard → archive chat
```

### Claude tool schema (server-side)

```ts
const submitProgramTool = {
  name: 'submit_program',
  description: 'Submit the finalized program once all questions are answered.',
  input_schema: {
    type: 'object',
    required: ['name', 'goals', 'weeksTotal', 'phases', 'days'],
    properties: {
      name: { type: 'string' },
      nameHe: { type: 'string' },
      description: { type: 'string' },
      goals: { type: 'array', items: { type: 'string' } },
      experienceLevel: { enum: ['beginner','intermediate','advanced'] },
      daysPerWeek: { type: 'number' },
      weeksTotal: { type: 'number', minimum: 1, maximum: 52 },
      phases: { type: 'array', items: {
        type: 'object',
        required: ['phase','weeks','tempo','restSeconds'],
        properties: {
          phase: { type: 'number' },
          weeks: { type: 'array', items: { type: 'number' }, minItems: 2, maxItems: 2 },
          tempo: { type: 'string' },
          restSeconds: { type: 'number' },
          focus: { type: 'string' },
        },
      } },
      days: { type: 'array', items: {
        type: 'object',
        required: ['day','title','exercises'],
        properties: {
          day: { type: 'number' },
          title: { type: 'string' },
          titleHe: { type: 'string' },
          exercises: { type: 'array', items: {
            type: 'object',
            required: ['libraryId','sets'],
            properties: {
              libraryId: { type: 'string' },     // MUST be in EXERCISE_LIBRARY
              sets: { type: 'number' },
              reps: { type: 'number' },
              durationSeconds: { type: 'number' },
              isUnilateral: { type: 'boolean' },
              isTimeBased: { type: 'boolean' },
              startWeakSide: { type: 'boolean' },
              notes: { type: 'string' },
              tag: { type: 'string' },
            },
          } },
        },
      } },
    },
  },
};
```

### Budget-aware chat (CRITICAL for cost control)

Free tier with strict budget. The chat itself must be aware of how many turns / tokens are left and **steer the user to finalize a program before the budget is exhausted**. The user must never be left without at least one program.

**Budget enforcement (server-side):**
- Hard cap: e.g. 30 user messages OR 200k total tokens per chat session, whichever comes first.
- Counter stored in `users/{uid}/limits/main`. Resets monthly.
- Per-month: e.g. 3 program-creation chats / month. After that, user can still **manually edit** existing programs but cannot start a new AI-driven chat until reset.

**Budget awareness (Claude's prompt):**
The system prompt to Claude includes the remaining budget on every turn:

```
You have ~12 turns left in this conversation before it auto-closes.
The user MUST leave this chat with a working program. If you're past
turn 8 and you still don't have enough info, START PROPOSING a sensible
program based on what you know and ask the user to confirm or tweak —
do not keep asking questions indefinitely.

When you have a reasonable program, call submit_program. The user can
always refine it later via manual editing in the app.
```

**Escape valve:**
- If a user hits the budget without a program saved, the server auto-generates a sensible starter program based on whatever chat data exists + sensible defaults, and saves it. Better a working starter than a locked-out user.
- The starter program is marked `isStarter: true` so the UI can prompt: "Refine this in 30 days when your chat budget resets, or edit it manually now."

**Manual editing path (no chat cost):**
- Once a user has at least one program, they can edit it freely in the UI (`<ProgramEdit />`) without any Claude calls.
- Edit operations: rename, reorder days, swap an exercise (from library), change sets/reps, add/remove a day, change phase tempo/rest.
- This is the always-available, zero-cost path. Chat is the convenience layer on top.

### Library injection & prompt caching

The library catalog is large (~100 items). Inject it once at the start of the system prompt with cache breakpoints:

```ts
messages: [{
  role: 'user',
  content: [
    {
      type: 'text',
      text: `Available exercises:\n${EXERCISE_LIBRARY_JSON}`,
      cache_control: { type: 'ephemeral' }, // cache this block
    },
    { type: 'text', text: userMessage },
  ],
}]
```

This gives ~90% cost reduction on repeated calls within the 5-min cache window. Critical for chat where multiple turns happen close together.

## 7. Backend Design

### Stack

- Firebase Cloud Functions (Node 20, TypeScript)
- Endpoints under `https://<region>-boostmind-b052c.cloudfunctions.net/api/...`
- Anthropic SDK (`@anthropic-ai/sdk`)
- Firebase Admin SDK (write to Firestore, verify auth)

### Region

**`me-west1` (Tel Aviv)** for both Cloud Functions and Firestore. Lowest latency for the target market (Israel / Middle East). Anthropic API still calls out to US — that's an extra ~150ms one-time per turn, acceptable for a chat UX.

### Endpoints

| Method | Path | Purpose | Streaming |
|---|---|---|---|
| `POST` | `/api/programs/chat` | Send user message, get Claude reply via SSE. | Yes (SSE) |
| `POST` | `/api/programs/generate` | Force-finalize current chat into a Program. Server validates + saves. | No |
| `POST` | `/api/programs/iterate` | Take an existing program + user feedback → generate a v2 via chat. | Yes (SSE) |
| `GET` | `/api/library` | Return the current EXERCISE_LIBRARY (server is source of truth). | No |
| `POST` | `/api/push/subscribe` | Register a Web Push subscription for the current user. | No |
| `POST` | `/api/push/schedule-rest` | Schedule a push to fire when rest timer ends. | No |
| `POST` | `/api/push/cancel-rest` | Cancel a pending scheduled rest push (user skipped or finished early). | No |

### Auth

Every endpoint verifies `Authorization: Bearer <firebase-id-token>` using Firebase Admin. Reject if missing/expired. Use `decoded.uid` for all writes.

### Rate limiting

In-memory or Firestore-backed counter, e.g.:

```ts
users/{uid}/limits/main → { chatMessagesToday: 42, programsThisMonth: 1, resetAt: ... }
```

Defaults:
- 50 chat messages / day / free user
- 5 program generations / month / free user
- 200ms minimum between requests
- Cooldown after rate-limit hit, return 429

### Secrets

- `ANTHROPIC_API_KEY` — Firebase Functions config (`firebase functions:secrets:set`)
- Never logged. Never returned to client.

### Cost model (rough)

Per program creation:
- ~5-10 chat turns
- Each turn: ~30k tokens input (catalog + history), ~500 tokens output
- With prompt caching: ~$0.05-0.15 per program
- Without caching: ~$0.50-1.50 per program

→ At 1000 free programs/month: $50-150 budget. Comfortable for free tier.

## 7b. Web Push for rest timer (PWA)

Today the rest timer breaks when the PWA is backgrounded on iOS (JS suspends, notification doesn't fire). Now that we'll have a server, solve it with real push notifications.

### Stack

- **VAPID keys** generated once, stored in Firebase Functions config
- **Service Worker** in the PWA handles `push` events and shows the notification
- **Push subscription** captured on first use, stored in `users/{uid}/pushSubscriptions/{id}`
- **Scheduled delivery** via Cloud Tasks (or a simple `setTimeout` in a long-running Function — Cloud Tasks is more reliable)

### Flow

```
1. User starts a rest timer for N seconds in the Workout screen.
2. Client POSTs /api/push/schedule-rest { restSeconds: N, message: "Set 3 ready" }
3. Server creates a Cloud Task with delay=N seconds.
4. After N seconds, task fires → server sends a Web Push to the user's
   subscription with the rest-complete payload.
5. Service Worker receives push → shows OS notification "Rest done — Set 3 ready".
6. If user skips/extends timer client-side, client POSTs /api/push/cancel-rest
   (or /api/push/schedule-rest with new time, which cancels the old task).
```

### iOS PWA notes

- iOS 16.4+ supports Web Push, but **only for PWAs added to the home screen**.
- User must explicitly grant notification permission. Trigger the prompt on first rest-timer start, not on app load.
- Notifications work even with the PWA fully closed or device locked.
- VAPID key registration must happen in the service worker.

### Fallback

If notification permission is denied or device doesn't support Web Push, fall back to current in-app behavior (timer keeps running, no OS notification). Already implemented.

Passcode model is **dropped**. From v1, every user goes through Firebase Auth:

- **Google sign-in** (one-tap on most devices, ideal for the Israel mobile market)
- **Email magic link** (fallback, no password to remember — UX-friendly)

One-time script. Shlomi signs in via Firebase Auth → captures his new `auth.uid` → migrates all `users/user_6724/*` Firestore data to `users/{newUid}/*`. Run with Firebase Admin SDK. After migration, the `user_6724` document tree is deleted.

This is a single-record migration — fine to do manually with a script + confirmation prompt.

### Firestore rules (locked down from day 1)

```
rules_version = '2';
service cloud.firestore {
  match /databases/{db}/documents {
    match /users/{uid}/{document=**} {
      allow read, write: if request.auth != null && request.auth.uid == uid;
    }
  }
}
```

No more `allow read, write: if true`.

## 12. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Claude generates an invalid program | Strict tool schema + server-side validation; retry with error feedback if invalid |
| Claude picks exercises not in library | Force `libraryId` enum constraint in tool schema; reject on validation |
| Costs balloon | Per-user rate limits + prompt caching + token budget per session |
| Existing users' data corrupted in migration | Run migration as a one-time script with backup snapshot of Firestore first |
| iOS PWA breaks chat (background suspension) | Non-streaming responses; refresh on resume |
| Hebrew/RTL rendering bugs in new screens | Audit every new screen with `dir="rtl"`, test with real Hebrew strings |
| Rate-limit abuse / spam accounts | Email verification gate + per-IP limits at the function layer |

## TL;DR

**Three big rocks** to make this a product:

1. **Move the program out of code** → Firestore + multi-program support (Phase 0).
2. **Stand up the backend + chat creator** → Cloud Functions in `me-west1` + Claude (streaming) + library-constrained tool output + budget-aware chat + Web Push (Phases 1-2-2b).
3. **Add Firebase Auth + Hebrew-only polish + tighten security** for first public users (Phases 4-5).

Phase 0 is mechanical and unlocks everything else. Start there.

- Hebrew-only, Middle East market, `me-west1` region
- Firebase Auth only (no passcode)
- Free tier with strict per-month chat budget; Claude is **budget-aware** and pushed to finalize before lock-out
- Streaming chat (SSE)
- Web Push for rest timer (solves iOS PWA limitation)
- One active program per user
- Program length decided by Claude based on chat
- Custom exercises stay (user can add mid-workout)
